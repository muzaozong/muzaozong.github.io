Java逃逸分析
发表于 2018-09-04| 更新于 2019-12-02|Java
字数总计: 1.6k|阅读时长: 7 分钟| 阅读量:

问题

问题1

Java新建的对象一定是分配在堆中吗？

问题2

如下程序会消耗多少内存，128m内存(-Xms128m -Xmx128m)下运行期间会发生gc(包括minor gc)吗？

参数

-XX:+DoEscapeAnalysis

Java在Java SE 6u23以及以后的版本中支持并默认开启了逃逸分析的选项

用途

逃逸分析是用来分析新对象使用范围并决定是否在Java堆上分配它的技术。

逃逸状态

  1. GlobalEscape(全局逃逸)， 即一个对象的引用逃出了方法或者线程。例如，一个对象存储在静态字段中的对象，或者存储在一个逃逸对象的域中，或者作为当前方法的结果。
  2. ArgEscape(参数级逃逸)，即在方法调用过程当中传递对象的应用给一个方法。这种状态可以通过分析被调方法的二进制代码确定。
  3. NoEscape(没有逃逸)，一个可以进行标量替换的对象，意味着可以从生成的代码中删除其分配。

测试

测试1(没有逃逸)

逃逸参数开启

-XX:+DoEscapeAnalysis -verbose:gc -XX:+PrintGCTimeStamps

多次执行，基本上都可以在5ms左右跑完，执行过程中没有产生任何gc

逃逸参数关闭

-XX:-DoEscapeAnalysis -verbose:gc -XX:+PrintGCTimeStamps

多次执行，一般需要60~100ms左右才能跑完，并且在执行过程中产生了两次minor gc

测试2(参数级逃逸)

逃逸参数开启

-XX:+DoEscapeAnalysis -verbose:gc -XX:+PrintGCTimeStamps

多次执行，基本在6ms左右执行完成，执行过程中没有产生任何gc

逃逸参数关闭

-XX:-DoEscapeAnalysis -verbose:gc -XX:+PrintGCTimeStamps

多次执行，基本在80~100ms内执行完成，并且在执行过程中产生了4次minor gc

测试3(全局逃逸)

逃逸参数开启

-XX:+DoEscapeAnalysis -verbose:gc -XX:+PrintGCTimeStamps

逃逸参数关闭

-XX:-DoEscapeAnalysis -verbose:gc -XX:+PrintGCTimeStamps

测试4(全局逃逸)

逃逸参数开启
-XX:+DoEscapeAnalysis -verbose:gc -XX:+PrintGCTimeStamps
中间会发生2次minor gc

逃逸参数关闭
-XX:-DoEscapeAnalysis -verbose:gc -XX:+PrintGCTimeStamps
中间会发生4次minor gc

测试5(全局逃逸)

逃逸参数开启

-XX:+DoEscapeAnalysis -verbose:gc -XX:+PrintGCTimeStamps

发生2次minor gc和1次full gc

逃逸参数关闭

-XX:-DoEscapeAnalysis -verbose:gc -XX:+PrintGCTimeStamps

发生2次minor gc和1次full gc

其他情况

  1. 如果分配的不是一个byte数组，而是一个类的话，那情况又会发生不同。
  2. 如果这个类中包含了一个数组，包含了多个数组。
  3. ……

以上情况试了之后，感觉整个人都迷了，彻底不知道栈上分配的发生的边界情况。

猜想&结论

  1. 逃逸分析算法中间的实现过程比较复杂，不只是简单地和『逃逸状态』中所叙述的那种，参考『测试3』和『测试4』可知，被当成函数返回值时也不一定会判定为逃逸成功。
  2. 对于那些存在大量逃逸对象的应用，逃逸分析可以极大地提高运行效率。但是相反，若应用中很少存在逃逸对象，则可能会因为逃逸分析算法而导致效率下降。
  3. java中新建的对象不一定就分配在堆中，如果满足NoEscape条件，则不再一定是分配到堆中，而是有可能是栈上分配。
